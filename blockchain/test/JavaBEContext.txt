Bối cảnh (Context):
Tôi đang xây dựng một hệ thống Hybrid DeFi AMM áp dụng mô hình Off-chain Signed Virtual Amplification (OSVA). Hệ thống này sử dụng chuẩn LP Token ERC20 (như Uniswap V2) nhưng có trượt giá cực thấp (như Uniswap V3) nhờ một hệ số khuếch đại thanh khoản ảo ($\alpha$) được tính toán off-chain và cấp chữ ký điện tử.
Hiện tại tôi đang chạy demo với các Custom Token tự deploy trên Testnet, do đó chúng sẽ không tồn tại trên sàn CEX (như Binance).  Nhiệm vụ (Task):Hãy đóng vai trò là một Senior Backend Developer / Web3 Integration Engineer. Hãy viết mã nguồn cho ứng dụng Backend bằng Java 17 và Spring Boot 3.x. Ứng dụng này đóng vai trò là "Decision Engine" (Động cơ ra quyết định) và "Oracle Signer" cho giao thức OSVA.Yêu Cầu Kiến Trúc & Công Nghệ:Ngôn ngữ/Framework: Java 17, Spring Boot 3.x.Web3 Integration: Sử dụng thư viện web3j để tương tác với blockchain và tạo chữ ký EIP-712.Lập lịch: Sử dụng @Scheduled của Spring để tạo CronJob thu thập dữ liệu định kỳ.Toán học: Bắt buộc sử dụng BigDecimal với MathContext(10, RoundingMode.HALF_UP) cho mọi phép tính tài chính.CÁC MODULE CHỨC NĂNG YÊU CẦU (Vui lòng viết code theo từng phần):Module 1: Cấu hình Demo Mode & Market Data Worker (Thu thập dữ liệu CEX)Cấu hình (application.yml):YAMLosva:
  market-data:
    demo-mode: true
    mock-sigma: 0.01 # 1% (Có thể đổi khi demo)
    mock-depth-factor: 1.0 # Giả lập thanh khoản bình thường
Worker Service: Chạy định kỳ (ví dụ: mỗi 10 giây) gọi public API của Binance (/api/v3/ticker/24hr và /api/v3/depth).Logic Try-Catch / Fallback (Bắt buộc):NẾU gọi API thành công: Tính toán Độ biến động lịch sử ($\sigma$) và Hệ số độ sâu sổ lệnh ($M_{depth}$).  NẾU gọi API thất bại (lỗi 400, 404 Invalid Symbol) HOẶC demo-mode == true: Bắt exception, log cảnh báo "Kích hoạt Fallback Data", và sử dụng mock-sigma, mock-depth-factor từ file config.Module 2: Web3 Worker (Thu thập State Blockchain)Sử dụng web3j để đọc trạng thái hợp đồng OSVAPool (đọc biến reserve0 và reserve1) thông qua RPC endpoint.Nhiệm vụ: Tính toán tỷ lệ chênh lệch của Pool để phục vụ cho việc tính Hệ số phạt/thưởng mất cân bằng ($M_{imb}$).  Module 3: Thuật Toán Ra Quyết Định (Decision Engine)Viết một Service nhận các tham số từ Module 1 và Module 2 để tính toán hệ số $\alpha$ bằng BigDecimal.Bước 1 - Rủi ro cơ sở: Tính $\delta_{base} = \sigma \cdot Z$ (với Z = 2.0). Nếu $\delta_{base} > 0.05$, trả về $\alpha = 0$ ngay lập tức.  Bước 2 - Hệ số mất cân bằng ($M_{imb}$): Áp dụng công thức phạt (nếu làm lệch thêm) hoặc thưởng (nếu kéo pool về 50/50) với hằng số $k = 1.5$.  Bước 3 - Tổng hợp: Tính $\delta_{final} = \delta_{base} \cdot M_{imb} \cdot M_{depth}$.  Bước 4 - Tính Alpha: Áp dụng công thức $\alpha = \frac{1}{\delta_{final}} - 1$. Sau đó ép kiểu về long (làm tròn xuống).Chốt chặn: Đảm bảo kết quả cuối cùng nằm trong khoảng [0, MAX_ALPHA] (với MAX_ALPHA = 100). Hàm này phải là Pure Function để dễ viết Unit Test.  Module 4: Chữ Ký EIP-712 (Signer Service)Đọc ORACLE_PRIVATE_KEY từ biến môi trường.Sử dụng StructuredDataEncoder và Sign.signMessage của web3j. (Lưu ý: tham số thứ 3 của signMessage bắt buộc là false).Sử dụng Text Blocks (""") của Java 17 để tạo chuỗi JSON chuẩn EIP-712.Cấu trúc Domain: name: "OSVA_Protocol", version: "1.0", chainId, verifyingContract.Cấu trúc Message (SwapRequest): user (address), tokenIn (address), amountIn (uint256), alpha (uint256), deadline (uint256).  Trả về chuỗi Hex Signature (r, s, v gộp lại thành chuỗi 65 bytes có tiền tố "0x").Module 5: REST API Endpoint (Controller)Tạo endpoint: GET /api/v1/osva/quoteTham số query: user, tokenIn, amountIn.Logic:Tính deadline = unix timestamp hiện tại + 60 giây.Lấy hệ số $\alpha$ từ Decision Engine.Gọi Signer Service tạo chữ ký.Response JSON:JSON{
